LogOS — Shared Backend Analysis

What deeper integration looks like across Core · Shepherd · Roundtable given the stack, data, and audience

What the Stack Enables

✔ In Use D1 Database

SQLite at the edge. Already stores 7,959 verses + 139K morphology records. Single source for all Bible data.

✔ In Use Vectorize

Semantic vector search (768-dim bge embeddings). 15.9K verse vectors. Can index ANY content type — Bible verses, Shepherd topics, Roundtable guides.

✔ In Use Workers AI

Llama 4 Scout for Q&A, Greek parsing, verse analysis. Free tier is generous. One binding covers the entire ecosystem.

⚡ Available AI Search (Managed RAG)

Cloudflare's managed RAG. Create one instance, upload docs from all three products, single search endpoint returns cross-product results. Costs neurons but no infrastructure.

⚡ Available Shared D1 Tables

Add new tables to existing D1: shepherd_topics, roundtable_guides, cross_product_index. One database serves all three.

⚡ Available Shared Auth

Cloudflare Access or simple JWT. One auth config protects all AI-cost endpoints across the ecosystem.

Integration Options

Option A: Shared Vectorize Index

Add Shepherd topic content + Roundtable guide content to the existing Vectorize index. A single semantic search across the hub returns Bible verses + parenting topics + sermon guides.

User Search Query
      │
      ▼
LogOS Core /api/verses/semantic
      │
      ├── Vectorize: "bible-verse-embeddings" (15.9K verse vectors)
      └── NEW: Vectorize: "logos-content" (verses + shepherd + roundtable)
                    │
                    ▼
      Results: [verse, topic, guide] sorted by relevance
✅ Pros

Unified search across all content. Shepherd/Roundtable don't need their own backends — just indexed metadata. Low dev cost.

⚠️ Cons

Shepherd content is already baked into static JSON — this adds a maintenance step to keep the vector index in sync.

Option B: Shepherd Gets API Endpoints on Core

Instead of Shepherd baking all 98 verses into static JSON, it fetches scripture via LogOS Core API at build time — or at runtime for dynamic features.

Current:  Shepherd builds with 98 verses baked into static JSON files
Future:   Shepherd calls LogOS Core /api/verses/reference at build time
          (or at runtime for real-time lookups, age-adaptation)

New endpoint: POST /api/shepherd/advice
  Input: {"issue": "anger", "ageTier": "grades-2-3", "childContext": "..."}
  Output: {"verses": [...], "lessonId": "anger", "parentingTip": "..."}
  AI: Workers AI adapts biblical wisdom to the child's age + situation
✅ Pros

Shepherd content is no longer stale at build time. AI-powered parenting advice per age tier. Single Bible data source.

⚠️ Cons

Shepherd loses offline capability. Needs auth on new endpoints. More complex than current static approach.

Option C: Cloudflare AI Search (Managed RAG)

Create one AI Search instance. Upload Bible study data, Shepherd topics, and Roundtable guides as documents. Single API endpoint searches ALL content. No Vectorize management, no embedding pipeline to maintain.

AI Search Instance: "logos-knowledge"
  ├── Docs: Bible study content (from Core)
  ├── Docs: Parenting topics + verses (from Shepherd)
  └── Docs: Sermon guides + series (from Roundtable)

Query: "What does the Bible say about fear?"
  │
  ▼

  [2] Topic: Fear & Anxiety (Shepherd) — Lesson for grades 2-3
  [3] Guide: Battle of the Mind (Roundtable) — Session 3
✅ Pros

Zero infrastructure. Single search across all content. No embedding code to maintain. Built-in reranking.

⚠️ Cons

Neuron costs per query. Can't customize the embedding model. Less control than Vectorize. Duplicates existing Vectorize index.

Option D: Lightweight — Verse API for Shepherd Build

Minimum viable integration. Shepherd keeps its static JSON but uses LogOS Core's `/api/verses/reference` endpoint as the source of truth for scripture text at build time. A build script fetches verse data from Core instead of parsing BTB files.

Shepherd Build Process:
  1. Read topic JSON (local, defines which verses are needed)
  2. Fetch verse text from LogOS Core /api/verses/reference (batch)
  3. Populate topic JSON with fetched text
  4. Build static SPA as before

Single source of truth for verse text = LogOS Core D1
Shepherd stays fully static at runtime
✅ Pros

Shepherd stays static. Single Bible data source. Low effort. No runtime dependencies.

⚠️ Cons

Build-time dependency on Core API. No runtime dynamic features.

Analysis by Dimension

Data Cohesion

Right now each product has its own copy of verse data:

  • Core — D1 database, canonical
  • Shepherd — 98 verses baked into static JSON files
  • Roundtable — Anchor verses baked into guide JSON files

Option D solves this for build-time. Option B solves it for runtime. Options A/C solve it for search.

AI Cost Exposure

Workers AI is pay-per-call. If Shepherd and Roundtable start making AI calls, costs multiply. A shared auth + rate-limiting layer on Core prevents one product from burning the budget. Keep all AI calls routed through LogOS Core where they can be monitored and throttled.

Audience Needs

The unified vision is "church members come to one place." But:

  • A parent looking for parenting help doesn't need Greek morphology — they need Shepherd topics surfaced when they search
  • A small group leader needs sermon guides linked to the Bible passages being studied
  • A Bible student needs deep search and Greek tools

This argues for discovery integration (cross-product search) more than data integration. Each product serves a different job; the hub helps people find the right product.

Recommendation

Start with Options A + D — then evaluate C

Option D (Verse API for Shepherd build) is the quickest win. Shepherd stops baking its own copy of verse data. At build time it fetches from LogOS Core. Single Bible data source, zero architectural change to either app.

Option A (Shared Vectorize index) extends the existing Vectorize to include Shepherd topics + Roundtable guides. The hub's semantic search then returns cross-product results. This doesn't require Shepherd or Roundtable to change — just an indexing pipeline.

Option C (AI Search) is worth evaluating after launch when there's traffic. It could replace Vectorize entirely if the cost/benefit works. For now, Vectorize is already built and paid for.

Evaluating Option E (MCP Gateway) — This is the most architecturally interesting path. It turns LogOS from a collection of apps into a platform that any AI agent can talk to. Hermes already uses MCP natively — wiring this in means TARS can query the Bible directly, Jules can pull verse context while building content, and future agents get instant access. Worth prototyping alongside Options A/D.

What This Unlocks

Single Bible Source

Shepherd and Roundtable both pull verse text from LogOS Core. One correction fixes all three.

Cross-Product Discovery

Search from the hub returns Bible verses, parenting topics, AND sermon guides. Users find what they need regardless of which product it lives in.

Products Stay Independent

Shepherd and Roundtable remain static SPAs. No runtime coupling. No shared deployment. The hub links + indexes, it doesn't control.

AI Sits Behind Auth

All AI calls route through LogOS Core where auth, rate limits, and monitoring live. Shepherd and Roundtable never make direct AI calls.
