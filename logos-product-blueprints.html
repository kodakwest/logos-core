LogOS Shepherd — What It Needs

Shepherd is a Gold MVP — 13 topics, 98 verses, 7 age tiers, full lesson system. It doesn't need a backend rebuild. It needs two things from LogOS Core:

1. Bible Search Access

Shepherd's Scripture Explorer (cross-topic verse browser) currently uses baked-in data. Route it to LogOS Core's search API instead — same search experience as the hub, but inside Shepherd's UI.

2. Build-Time Verse Pipeline

At build time, fetch verse text from LogOS Core instead of parsing BTB files. Single source of truth for all Bible text. No change to Shepherd's runtime architecture.

Verdict: Shepherd stays a static SPA. It calls Core APIs at build time. The only runtime change is the Scripture Explorer which queries Core's search endpoint instead of local data.

Core Features

Sermon Knowledge Graph

  • 📖 Sermon → Verse — Every sermon cross-referenced to the Bible passages it covers
  • 📺 Sermon → YouTube — Video embeds, timestamps linked to outline sections
  • 🎧 Sermon → Spotify — Episode player, audio sections
  • 🔗 Sermon → Series — Full series navigation, chronological ordering
  • 🧠 Semantic search — Find sermons by topic, theme, or passage
  • 📊 Topic clustering — Auto-group sermons by theme across series

Interactive Guides + Gamification

  • 📝 Discussion forms — Answer questions inline, save responses per session
  • ✅ Progress tracking — Series completion, section-by-section check-off
  • 🏆 Achievements — "Completed the Battle of the Mind series", "Answered every question"
  • 👥 Group mode — Share responses with your small group
  • 📅 Session scheduling — "Next meeting: Wednesday" with guide pre-loaded
  • 🎨 Tasteful UI — Progress rings, subtle animations, no cheesy graphics

The Glue: LogOS MCP Gateway

An MCP server deployed as a Cloudflare Worker that exposes the entire ecosystem as tools any AI agent can call. This is the architectural layer that ties everything together without coupling the products.

The MCP Gateway doesn't replace the REST API — it wraps it. REST stays for frontend apps. MCP is for AI agents. Same backend, two protocols, one auth boundary.

How It All Connects

Roundtable is the product that needs the most architecture

It's not just a guide viewer — it's a sermon knowledge system with a graph database, vector search, media cross-referencing, AI-powered content generation, and gamified interactive sessions. That warrants its own Workers backend (same pattern as Core).

Shepherd stays light — static SPA that calls Core's search API at runtime for Scripture Explorer, and fetches verse text from Core at build time.

Core stays the data foundation — auth, verse data, AI, and the search layer that Shepherd and Roundtable both consume.

MCP Gateway wraps all three so any AI agent (TARS, Jules, future tools) can query the entire ecosystem through one protocol. No REST endpoints to learn, no auth headers to manage — just MCP tool discovery.
