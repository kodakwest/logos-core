LogOS · API Landscape

Current State → Future Architecture
● Current: Bible AI Search → Transition: API Enhancements ● Future: LogOS Ecosystem

Current State

  • 7,959 verses ingested (KJV + BSB + Greek NT)
  • 139,476 morphology records (lemma, POS, parsing)
  • 9 endpoints — mixed public/private on same Worker
  • llama-4-scout-17b for AI synthesis
  • bge-base-en-v1.5 for 768-dim embeddings
  • ⚠ ZERO auth on any endpoint — critical blocker
  • ⚠ Serial upload — D1, AI, Vectorize can drift

Future — LogOS

  • Auth Gateway — Cloudflare Access or JWT on AI-cost endpoints
  • Endpoint split — public-read / AI-cost / admin (separated by auth level)
  • New endpoints — GET /api/verses/:ref, GET /api/verses/chapter, paginated search
  • Batch ingestion — queued, resumable, with content hashes + embedding versioning
  • FTS Greek search — normalized token table replaces LIKE '%...%'
  • Rate limiting — per-key, per-endpoint budgets

Ecosystem Additions

  • Hub Portal — central navigation: Bible study, search, Q&A, ecosystem links
  • Shepherd API — Scripture Explorer consuming Core API at build + runtime
  • Roundtable Backend — new Worker + D1 + Vectorize for sermon knowledge system
  • MCP Gateway — one Worker exposing all LogOS tools for Hermes agents
  • gslogimaker pattern — dynamic verse rendering with togglable translation layers
  • Content pipeline — NA28 XML ingestion, automated sermon mapping
